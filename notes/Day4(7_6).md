# 基于余额连续性的交易流重构与社区端点容错合并方法

## 1 引言

在银行交易数据的分析过程中，单账户的交易记录通常以流水表形式存储，包含交易金额、交易后余额、收付标志等字段。然而，由于数据记录方式差异（如借贷方向与金额符号的不一致）、跨行结算产生的小额尾差等因素，直接依据原始字段进行余额连续性匹配往往导致账户交易链被割裂为多条互不相交的片段。本文提出一种基于余额连续性的交易流重构方法，并引入社区端点容错合并机制，在可容忍误差范围内将被割裂的交易片段重新连接为完整的交易链。

## 2 问题定义与数学模型

### 2.1 交易记录的形式化表示

设单账户的交易记录集合为 $T = \{t_1, t_2, \ldots, t_n\}$，每条记录 $t_i$ 包含以下关键字段：

- $a_i$：原始有符号交易金额（单位：分）
- $b_i$：交易后余额（单位：分）
- $d_i$：收付方向标识（进/出等）

定义方向乘子函数 $\delta(d_i)$：

$$
\delta(d_i) = \begin{cases}
+1, & d_i \in \{\text{进}, \text{收}, \text{贷}, \text{cr}, \text{credit}, \ldots\} \\
-1, & d_i \in \{\text{出}, \text{支}, \text{借}, \text{dr}, \text{debit}, \ldots\}
\end{cases}
$$

### 2.2 余额影响额

定义**余额影响额** $e_i$ 为交易 $t_i$ 对账户余额的实际影响值：

$$
e_i = \delta(d_i) \times a_i
$$

根据会计恒等式，交易前余额 $b_i^{\text{before}}$ 与交易后余额 $b_i^{\text{after}}$ 满足：

$$
b_i^{\text{before}} = b_i^{\text{after}} - e_i
$$

其中 $b_i^{\text{after}} = b_i$ 即为 CSV 中记录的余额字段。

### 2.3 余额连续性约束

对于同一账户中的任意两条交易 $t_i$ 和 $t_j$，若 $t_i$ 是 $t_j$ 的直接前驱，则必须满足余额连续性条件：

$$
b_i^{\text{after}} = b_j^{\text{before}}
$$

即前一条交易的结束余额等于后一条交易的起始余额。此约束是交易流重构的核心依据，不依赖于交易记录的物理行序或时间戳。

## 3 交易流重构方法

### 3.1 候选边生成

对于给定的交易集合 $T$，定义前驱节点集合 $S$ 和后继节点集合 $U$：

$$
S = \{i \mid b_i^{\text{after}} = v, i \in T\}, \quad U = \{j \mid b_j^{\text{before}} = v, j \in T\}
$$

对于每个余额值 $v$，候选边集合 $C_v$ 定义为：

$$
C_v = \{(i, j) \mid i \in S_v, j \in U_v, i \neq j\}
$$

当 $|S_v| \times |U_v|$ 超过预设阈值（默认 250,000）时，采用稀疏化策略：对每个节点保留时间邻近的候选边，同时通过最大基数骨架边保证匹配的完整性。

每条候选边 $(i, j)$ 的代价函数 $c(i, j)$ 定义为：

$$
c(i, j) = c_{\text{time}}(i, j) + c_{\text{row}}(i, j) + c_{\text{conflict}}(i, j)
$$

其中 $c_{\text{time}}$ 基于时间差的对数惩罚，$c_{\text{row}}$ 基于行号差的线性惩罚，$c_{\text{conflict}}$ 为方向冲突的额外代价。

### 3.2 最大基数最小代价匹配

在候选边集合上，采用两阶段匹配策略：

**阶段一**（Hopcroft-Karp 算法）：在二分图中求解最大基数匹配，确定匹配基数的理论上界 $k_{\max}$。

**阶段二**（最小费用最大流）：在保证匹配基数达到 $k_{\max}$ 的约束下，最小化匹配的总代价。构建流网络 $G = (V, E)$，其中源点 $s$ 连接所有左节点，所有右节点连接汇点 $t$，边容量均为 1，费用为 $c(i, j)$。通过连续最短增广路算法求解。

当问题规模超过阈值（节点数 > 5,000 或边数 > 200,000）时，仅执行阶段一以降低计算复杂度。

### 3.2.1 Hopcroft-Karp 算法详解

#### 问题定义

给定二分图 $G = (U, V, E)$，其中 $U$ 为左部节点集合（表示交易的前驱候选），$V$ 为右部节点集合（表示交易的后继候选），$E \subseteq U \times V$ 为候选边集合。二分图匹配的目标是选择边子集 $M \subseteq E$，使得每个节点在 $M$ 中至多出现一次。**最大基数匹配**是指 $|M|$ 最大的匹配。

#### 增广路径

给定当前匹配 $M$，一条**增广路径**是满足以下条件的路径：

1. 起点和终点都是未匹配节点（一端在 $U$、另一端在 $V$）
2. 路径上的边交替属于 $E \setminus M$（未匹配边）和 $M$（已匹配边）
3. 路径长度（边数）为奇数

对增广路径执行**取反操作**（将路径上未匹配边变为匹配边、已匹配边变为未匹配边），可以使匹配基数增加 1。Berge 引理指出：**匹配 $M$ 是最大基数匹配当且仅当 $G$ 中不存在关于 $M$ 的增广路径。**

#### 朴素算法的瓶颈

最直接的匹配算法是重复执行以下步骤：DFS 搜索一条增广路径 → 取反 → 重复。每次增广需要 $O(|E|)$ 时间，最多增广 $O(|V|)$ 次，总时间复杂度为 $O(|V| \cdot |E|)$。当图中节点数达到数千、边数达到数十万时（如本应用场景），此复杂度不可接受。

#### Hopcroft-Karp 的核心思想

Hopcroft-Karp 算法通过**同时寻找多条不相交的增广路径**来加速。其时间复杂度为 $O(|E| \sqrt{|V|})$，远优于朴素算法的 $O(|V| \cdot |E|)$。算法由两个阶段交替执行：

**阶段一（BFS 构建层次图）**：从所有未匹配的左部节点出发，沿交替路径（未匹配边 → 已匹配边 → 未匹配边 → ...）进行广度优先搜索，计算每个节点到起点的距离。当到达某个未匹配的右部节点时，标记"找到增广路径"。BFS 终止后，所有距离值构成了一个层次图。

**阶段二（DFS 沿层次图扩展匹配）**：对每个未匹配的左部节点，在层次图约束下执行深度优先搜索，寻找增广路径。关键约束是 `dist[mate] == dist[u] + 1`——只允许沿层次递增的方向前进，确保 DFS 找到的是**最短路**增广路径。每次找到增广路径后立即执行取反操作。

一轮 BFS + 多轮 DFS 称为一个**阶段**。每阶段可以找到多条**节点不相交**（顶点不重复）的最短增广路径，使匹配基数增加的数量等于该轮找到的路径数。

#### 算法正确性

Hopcroft-Karp 的正确性建立在以下性质之上：

1. **最短路增广性质**：每轮 BFS 构建的层次图保证 DFS 只寻找当前最短的增广路径
2. **不相交性质**：一轮中找到的多条增广路径节点互不相交，因此可以独立执行取反操作
3. **层次递增性质**：每轮之后最短增广路径的长度严格增加
4. **终止性质**：经过 $O(\sqrt{|V|})$ 轮后必达到最大匹配

#### 代码实现对照

```python
def hopcroft_karp(n, adjacency):
    pair_u = [-1] * n      # pair_u[u] = v: 左节点 u 匹配到 v; -1 表示未匹配
    pair_v = [-1] * n      # pair_v[v] = u: 右节点 v 匹配到 u
    dist = [0] * n         # BFS 层次距离
    inf = 10**18

    def bfs() -> bool:
        """BFS 构建层次图，检测是否存在增广路径。"""
        queue = deque()
        found = False
        for u in range(n):
            if pair_u[u] == -1:
                dist[u] = 0          # 未匹配起点距离为 0
                queue.append(u)
            else:
                dist[u] = inf         # 已匹配节点初始化为无穷
        while queue:
            u = queue.popleft()
            for v in adjacency.get(u, ()):
                mate = pair_v[v]      # v 的当前匹配
                if mate == -1:
                    found = True      # 发现未匹配的右部节点 → 有增广路径
                elif dist[mate] == inf:
                    dist[mate] = dist[u] + 1  # 沿已匹配边前进到下一层
                    queue.append(mate)
        return found

    def dfs(u: int) -> bool:
        """DFS 沿层次图寻找一条增广路径并立即取反。"""
        for v in adjacency.get(u, ()):
            mate = pair_v[v]
            if mate == -1 or (dist[mate] == dist[u] + 1 and dfs(mate)):
                pair_u[u] = v
                pair_v[v] = u
                return True           # 取反成功，匹配基数 +1
        dist[u] = inf                  # 标记为死路，后续不再访问
        return False

    matching = 0
    while bfs():                       # 外层循环：每轮 BFS 构建层次图
        for u in range(n):
            if pair_u[u] == -1 and dfs(u):
                matching += 1           # 内层循环：每找到一条增广路径就 +1
    return pair_u, pair_v, matching
```

关键设计点说明：

- **`pair_u` / `pair_v` 分离**：将左部和右部的匹配关系分开存储，便于快速查询任意侧节点的匹配状态
- **BFS 只从未匹配左节点出发**：保证搜索的起始点符合增广路径的定义
- **`dist[mate] = dist[u] + 1`**：BFS 沿已匹配边前进一层，DFS 沿此约束确保只走最短路
- **`dist[u] = inf`（DFS 中）**：当某个节点无法找到增广路径时标记为无穷，后续轮次不再重复搜索

#### 算法示例

假设初始匹配为空，图结构如下：

```text
左部 U: {u0, u1, u2, u3}
右部 V: {v0, v1, v2, v3}
边:     u0→v0, u0→v1, u1→v2, u2→v2, u3→v3
```

**第 1 轮 BFS**：

```text
起点 (未匹配左部): u0(dist=0), u1(dist=0), u2(dist=0), u3(dist=0)

u0 → v0(未匹配→found), v1(未匹配→found)
u1 → v2(未匹配→found)
u2 → v2(未匹配→found)
u3 → v3(未匹配→found)
```

由于当前无匹配边，BFS 中不会走 `dist[mate] = dist[u] + 1` 的分支。BFS 直接发现所有右部节点都未匹配。

**第 1 轮 DFS**（层次距离均为 0，每个左节点尝试找一条路径）：

- `u0 → v0`：v0 未匹配 → 匹配 `(u0, v0)`，matching=1
- `u1 → v2`：v2 未匹配 → 匹配 `(u1, v2)`，matching=2
- `u2 → v2`：v2 已匹配给 u1，检查 `dist[u1] == dist[u2] + 1`？u2 的 dist 已被 BFS 设为 0（未匹配），不符合 → 此路不通
- `u3 → v3`：v3 未匹配 → 匹配 `(u3, v3)`，matching=3

结果：匹配基数 3，边为 `{(u0,v0), (u1,v2), (u3,v3)}`。

**第 2 轮 BFS**：

```text
起点 (未匹配左部): u2(dist=0)

u2 → v2(已匹配给 u1) → mate=u1 → dist[u1]=dist[2]+1=1, 入队
u1 → v?(只有 v2), 已匹配 → mate=v2, 入队 ... 但 u2 无法到达未匹配右节点
```

无到达未匹配右节点的路径 → BFS 返回 False → 算法终止。

最终匹配基数 = 3，达到了此图的最大基数（左部节点全被匹配）。

#### 在本项目中的角色

`hopcroft_karp` 函数的返回值 `pair_u` 会被 `choose_matching_for_size` 用于以下两个场景之一：

1. **直接作为匹配结果**（大规模图）：当节点数 > 5,000 或边数 > 200,000 时，HK 的结果直接作为最终匹配，不再执行 min-cost flow
2. **确定基数上界**（小规模图）：`matching` 返回值作为 `required_flow` 参数传入 `min_cost_matching_exact_cardinality`，告诉最小费用流"必须匹配这么多条边"

```python
pair_u, _, cardinality = hopcroft_karp(n, adjacency)  # 求最大基数
if cardinality == 0:
    return {}, "NO_EDGES"

if n <= exact_weight_node_limit and len(edges) <= exact_weight_edge_limit:
    return (
        min_cost_matching_exact_cardinality(n, edges, cardinality),  # 基数硬约束
        "EXACT_MIN_COST_MAXIMUM_MATCHING",
    )

return ({u: v for u, v in enumerate(pair_u) if v != -1}, "FAST_MAXIMUM_MATCHING")
```

### 3.3 初始流分解

匹配结果表示为后继映射 $\phi: I \to I$，其中 $I = \{0, 1, \ldots, n-1\}$ 为交易索引集合。$\phi(i) = j$ 表示交易 $i$ 是交易 $j$ 的直接前驱。

从前驱映射 $\psi(j) = i$ 出发，定义路径起点集合：

$$
I_{\text{start}} = \{i \in I \mid \psi(i) = -1\}
$$

从每个起点出发，沿 $\phi$ 指针追踪得到有序交易序列，构成**开放路径**（有向路径）。剩余节点必然位于若干个有向环中，提取后构成**循环路径**（哈密顿环）。

每条路径或环表示为一个数值社区 $S_k = (I_k, \text{isCycle}_k)$，其中 $I_k$ 为有序的交易索引序列。

## 4 社区端点容错合并方法

### 4.1 问题动机

经严格余额连续性匹配后，若账户交易记录中存在小额金额误差（如手续费、跨行结算尾差）或收付方向符号问题（如借贷方向与金额符号不一致），则余额连续性条件 $b_i^{\text{after}} = b_j^{\text{before}}$ 无法严格满足，导致原本连续的交易链被割裂为多条开放路径。社区端点容错合并旨在可容忍误差范围内，将这些开放路径首尾相连。

### 4.2 修复候选的数学定义

设存在两条开放路径 $P_s$ 和 $P_t$，其中：

- $B_s = b^{\text{after}}( \text{tail}(P_s) )$：路径 $P_s$ 末节点的交易后余额
- $B_t = b^{\text{after}}( \text{head}(P_t) )$：路径 $P_t$ 首节点的交易后余额
- $E_t = e( \text{head}(P_t) )$：路径 $P_t$ 首节点的当前余额影响额

为使两条路径连续，$P_t$ 首节点所需的理想余额影响额 $E^*$ 唯一确定为：

$$
E^* = B_t - B_s
$$

定义两种误差度量：

- **普通金额误差**：$\varepsilon_{\text{normal}} = E^* - E_t$
- **反号误差**：$\varepsilon_{\text{opposite}} = E^* + E_t$（对应于 $E_t$ 取反后的场景）

当 $\varepsilon_{\text{normal}} = 0$ 或 $|\varepsilon_{\text{normal}}| < \tau$ 时，称为**普通模式修复**；当 $\varepsilon_{\text{opposite}} = 0$ 或 $|\varepsilon_{\text{opposite}}| < \tau$ 时（且修复开启），称为**符号修复模式**。其中 $\tau$ 为容差阈值（默认 100 分，即 ¥1.00，采用严格小于比较）。

### 4.3 修复类型与优先级

根据误差性质，修复候选可分为四类，按优先级递减排列：

| 优先级 | 修复类型 | 判定条件 |
|--------|----------|----------|
| 0 | 精确端点恢复（Exact Endpoint Recovery） | $\varepsilon_{\text{normal}} = 0$ |
| 1 | 小额金额误差（Small Amount Error） | $0 < \|\varepsilon_{\text{normal}}\| < \tau$ |
| 2 | 符号问题（Sign Error） | $\varepsilon_{\text{opposite}} = 0$ |
| 3 | 符号问题兼小额误差（Sign + Small Amount） | $0 < \|\varepsilon_{\text{opposite}}\| < \tau$ |

### 4.4 候选代价函数

修复候选的代价函数采用三层数量级互不重叠的编码方案：

$$
c_{\text{bridge}} = r \times 10^{15} + |\varepsilon_{\text{ref}}| \times 10^{9} + \min(c_{\text{time}}, 10^{8})
$$

其中 $r \in \{0, 1, 2, 3\}$ 为修复类型的优先级秩，$\varepsilon_{\text{ref}}$ 为实际使用的参考误差（普通模式使用 $\varepsilon_{\text{normal}}$，符号修复模式使用 $\varepsilon_{\text{opposite}}$），$c_{\text{time}}$ 为基于时间差的行号差的附加代价。

该设计确保匹配过程严格遵循**修复类型优先 → 误差绝对值次之 → 时间代价再次之**的排序规则。

### 4.5 搜索优化与去重

对所有开放路径的末节点，按余额值 $b^{\text{after}}$ 建立哈希索引。对每个候选目标的首节点所需余额 $B_t - E_t$（或 $B_t + E_t$），通过二分查找在 $O(\log m)$ 时间内定位容差范围内的候选源节点，避免 $O(m^2)$ 的暴力枚举。

去重策略分为两层：

1. **同目标去重**：对同一目标路径，相同的源-目标对可能因普通模式和符号修复模式同时满足条件而产生多个候选，按代价三元组 $(c_{\text{bridge}}, |\varepsilon_{\text{ref}}|, \text{source\_no})$ 保留最优
2. **全局去重**：不同目标路径可能产生相同的源-目标对，保留代价最小的候选

### 4.6 最优合并方案的选择

将修复候选选择问题建模为二分图上的最大基数最小代价匹配，其中节点为开放路径，边为修复候选。求解框架与第 3.2 节相同，即 Hopcroft-Karp 算法求最大基数，辅以最小费用最大流（小规模）或纯 Hopcroft-Karp（大规模）求最优解。

### 4.7 防环机制

容错修复基于金额近似匹配而非严格相等，可能将本不相关的开放路径凭数值巧合连接为闭环，产生虚假的"全账户大闭环"。为此引入防环机制：

对有向图 $G_{\text{repair}} = (V_{\text{stream}}, E_{\text{selected}})$ 采用三色深度优先搜索检测有向环，对每个检测到的环删除其中代价最高（即最不可信）的修复边：

$$
e_{\text{remove}} = \arg\max_{e \in E_{\text{cycle}}} (c_{\text{bridge}}(e), |\varepsilon_{\text{ref}}(e)|, \text{source\_no}(e))
$$

此机制默认启用，可通过参数 `--allow-repair-cycles` 关闭。

### 4.8 合并操作的底层实现

选中的修复边通过以下操作应用到后继映射 $\phi$：

$$
\phi(\text{source\_node}) = \text{target\_node}
$$

同时更新目标节点的余额影响额：

$$
e(\text{target\_node}) = E^*
$$

此操作的本质是在有向链表 $\phi$ 中追加一条边。合并前，两条开放路径在 $\phi$ 中分属两个连通分量；合并后，由于新增边的连接，两个分量合并为一个连通分量。随后重新执行流分解算法即可得到合并后的完整路径。

该操作的时间复杂度为 $O(1)$，无需节点复制或数据迁移。

## 5 算法流程总结

社区端点修复的整体流程可概括为以下步骤：

1. **严格匹配阶段**：基于余额连续性约束构建候选边集合，通过最大基数最小代价匹配获得初始后继映射 $\phi$，分解得到初始数值社区集合 $\{S_k\}$
2. **修复候选构建**：遍历所有开放路径对，计算每种连接方案的误差度量，在容差阈值内生成修复候选集合 $\mathcal{B}$
3. **最优方案选择**：以开放路径为节点、修复候选为边构建二分图，求解最大基数最小代价匹配
4. **防环处理**：在选中的修复边构成的有向图中检测并解除闭环
5. **合并执行**：将选中的修复边追加到后继映射 $\phi$，更新目标节点的余额影响额
6. **最终分解**：在更新后的后继映射上重新执行流分解，输出合并后的完整交易链

## 6 输出与验证

合并结果输出为以下文件：

- **路径/环 CSV**：包含每个节点的原始字段、修正字段、余额连续性指标
- **社区边界修复诊断 CSV**：记录每个修复候选的误差度量、修复类型、是否被选中
- **SVG 流图**：可视化交易链，以不同颜色标识修复边（橙色表示金额修复、紫色表示符号修复）
- **HTML 浏览索引**：支持缩放平移的交互式浏览界面

所有修正仅作用于推断值（余额影响额），不覆盖原始 CSV 数据，保证数据可追溯性。

## 7 端点修复算法逐步骤详解

### 7.1 整体数据流

端点修复算法在 `process_file` 函数中位于严格匹配和最终流分解之间，其数据流如下：

```text
┌─────────────────────────────────────────────────────────────┐
│ 步骤 A：严格匹配                                            │
│   build_candidate_edges → choose_matching → decompose_streams │
│   结果：successor（严格匹配边）、initial_streams（初始社区）   │
└────────────────────────────────┬────────────────────────────┘
                                 ↓
┌─────────────────────────────────────────────────────────────┐
│ 步骤 B：社区端点修复（仅在 --merge-community-boundaries 开启）│
│                                                              │
│  B1. build_community_bridges(initial_streams)                 │
│      → 遍历所有的开放路径对，生成 CommunityBridge 候选列表     │
│                                                              │
│  B2. choose_community_bridges(candidates)                     │
│      → 以流为节点、候选为边，做最大基数最小代价匹配             │
│      → 对选中的边做防环删除                                   │
│                                                              │
│  B3. apply_community_bridges(successor, selected)             │
│      → 在 successor 中追加修复边                              │
│      → 更新 inferred_effects                                  │
└────────────────────────────────┬────────────────────────────┘
                                 ↓
┌─────────────────────────────────────────────────────────────┐
│ 步骤 C：最终分解                                             │
│   decompose_streams(successor)  ← 此时 successor 含严格+修复边│
│   结果：合并后的完整流                                        │
└─────────────────────────────────────────────────────────────┘
```

### 7.2 候选构建（build_community_bridges）的完整执行流程

#### 第 1 步：筛选开放路径

```python
open_indices = [
    i for i, stream in enumerate(streams)
    if stream.node_indices and not stream.is_cycle
]
```

只有**开放路径**（非环）才参与修复。环已形成闭环，不需要也**不应该**被连接。

假设经过严格匹配后得到 3 条开放路径和 1 个环：

```text
社区 1（开放路径）: [A] → [B] → [C]    末节点 C: after=100
社区 2（开放路径）: [D] → [E]          末节点 E: after=150
社区 3（开放路径）: [F] → [G] → [H]    末节点 H: after=90
社区 4（环）:       [I] → [J] → [K] → [I]   ← 不参与修复
```

`open_indices = [0, 1, 2]`（社区 4 被排除）。

#### 第 2 步：建立余额索引

```python
tails_by_balance: Dict[int, List[int]] = defaultdict(list)
for stream_index in open_indices:
    tail = streams[stream_index].node_indices[-1]
    tails_by_balance[transactions[tail].after_minor].append(stream_index)
sorted_balances = sorted(tails_by_balance)
```

结果：

```text

    100: [0],     # 社区 1 的末节点 C: after=100
    150: [1],     # 社区 2 的末节点 E: after=150
    90:  [2],     # 社区 3 的末节点 H: after=90
}
sorted_balances = [90, 100, 150]
```

#### 第 3 步：外层循环——以每条开放路径为 target

```python

    target_stream = streams[target_stream_index]
    target_node = target_stream.node_indices[0]  # 首节点
    target_tx = transactions[target_node]
    original_effect = inferred_effects[target_node]  # E_t
```

对社区 2（作为 target），首节点是 D：

```text
target_tx = D
B_t = D.after = 150
E_t = D.effective = 50（假设）
```

#### 第 4 步：计算 target 所需的"前驱末节点余额"

```python
centers = [(False, target_tx.after_minor - original_effect)]
if allow_sign_repair and original_effect != 0:
    centers.append((True, target_tx.after_minor + original_effect))
```

社区 2 作为 target：

```text
普通模式 (use_opposite=False):
  center = B_t - E_t = 150 - 50 = 100
  → 只要找到 after=100 的末节点，社区 2 就可以连在后面

符号修复模式 (use_opposite=True):
  center = B_t + E_t = 150 + 50 = 200
  → 若 E_t 取反后匹配，所需前驱末余额为 200
```

#### 第 5 步：内层循环——搜索 source 末节点

```python
    lo = bisect.bisect_left(sorted_balances, center_balance - radius)
    hi = bisect.bisect_right(sorted_balances, center_balance + radius)
    for source_balance in sorted_balances[lo:hi]:
        for source_stream_index in tails_by_balance[source_balance]:
```

以普通模式 center=100 为例，tolerance=100（¥1.00），radius=99：

```text
lo = bisect_left([90, 100, 150], 100 - 99) = bisect_left([90, 100, 150], 1) = 0
hi = bisect_right([90, 100, 150], 100 + 99) = bisect_right([90, 100, 150], 199) = 2
→ 搜索余额范围：sorted_balances[0:2] = [90, 100]
→ 遍历这些余额下的 source 末节点
```

- 余额=90：社区 3 的末节点 H（after=90）
- 余额=100：社区 1 的末节点 C（after=100）

社区 2 不与自身连接（`if source == target: continue`）。

#### 第 6 步：误差计算

```python
desired_effect = target_tx.after_minor - source_tx.after_minor
normal_error = desired_effect - original_effect
opposite_error = desired_effect + original_effect
reference_error = opposite_error if use_opposite else normal_error
```

社区 1（source=C, after=100）→ 社区 2（target=D, after=150, E_t=50）：

```text
desired_effect  = B_t - B_s = 150 - 100 = 50    # 理想余额影响 E*
normal_error    = 50 - 50 = 0                     # 精确匹配！
opposite_error  = 50 + 50 = 100                   # 取反误差 100 分
reference_error = 0 (普通模式)
```

社区 3（source=H, after=90）→ 社区 2（target=D, after=150, E_t=50）：

```text
desired_effect  = 150 - 90 = 60
normal_error    = 60 - 50 = 10                    # 小额误差 10 分（¥0.10）
opposite_error  = 60 + 50 = 110
reference_error = 10 (普通模式)
```

#### 第 7 步：阈值判断

```python
allowed = reference_error == 0 or (
    tolerance_minor > 0 and abs(reference_error) < tolerance_minor
)
```

以默认 tolerance=100（¥1.00）判断：

| 候选 | source | target | reference_error | 允许? | 原因 |
|---|---|---|---|---|---|
| b₁ | 社区1尾(after=100) | 社区2首(D) | 0 | ✓ | 精确端点恢复 |
| b₂ | 社区3尾(after=90) | 社区2首(D) | 10 | ✓ | 10 < 100，小额误差 |
| b₃ | 社区1尾(after=100) | 社区3首(F) | ... | 计算... | 取决于 F 的数据 |

#### 第 8 步：修复类型与代价

```python
repair_type = _bridge_repair_type(normal_error, opposite_error, use_opposite)
class_rank = {"EXACT_ENDPOINT_RECOVERY": 0, "SMALL_AMOUNT_ERROR": 1, ...}[repair_type]
cost = class_rank * 10**15 + abs(reference_error) * 10**9 + min(time_cost, 10**8)
```

b₁：class_rank=0, |ref_error|=0 → cost ≈ 0（精确恢复，代价极低）
b₂：class_rank=1, |ref_error|=10 → cost ≈ 1×10¹⁵ + 10×10⁹ ≈ 1.00000001×10¹⁵

#### 第 9 步：构造 CommunityBridge 对象

```python
bridge = CommunityBridge(
    source_node=C的索引,          # 社区1末节点
    target_node=D的索引,          # 社区2首节点
    repair_type="EXACT_ENDPOINT_RECOVERY",
    original_effect_minor=50,     # D 原始余额影响
    corrected_effect_minor=50,    # E* = 50（与原始一致，无需修正）
    normal_error_minor=0,         # 精确匹配
    ...
)
```

对于 b₁（精确恢复），`corrected_effect_minor=desired_effect=50` 等于 `original_effect=50`，实际上不需要修正。但对于有误差的情况，`corrected_effect_minor` 会被设为 `desired_effect`，覆盖原始值。

#### 第 10 步：同 target 去重

```python

previous = per_target.get(key)
if previous is None or (cost, |error|, source_no) < (prev_cost, prev_error, prev_source):
    per_target[key] = bridge
```

如果同一个 (source, target) 对同时满足普通模式和符号修复模式（例如普通模式误差 3 分、符号模式误差 2 分），保留代价较小的那个。

#### 第 11 步：截断和全局去重

```python
ranked = sorted(per_target.values(), key=lambda b: b.cost)
if max_candidates_per_start > 0:
    ranked = ranked[:max_candidates_per_start]
for bridge in ranked:

```

每个 target 最多保留 `max_candidates_per_start`（默认 128）个候选，然后全局去重。

### 7.3 匹配选择（choose_community_bridges）的完整流程

#### 7.3.1 匹配的对象是"流"而非"交易"

严格匹配阶段：
- 二分图节点 = **交易索引**（每个交易是一个节点）
- 边 = 候选边（`CandidateEdge`）
- 匹配结果 `successor: Dict[int, int]`（交易索引 → 交易索引）

社区修复阶段：
- 二分图节点 = **流索引**（每条开放路径是一个节点）
- 边 = 修复候选（`CommunityBridge`）
- 匹配结果 `selected_pairs: Dict[int, int]`（流索引 → 流索引）

```text
严格匹配:  交易0 → 交易5, 交易3 → 交易7, ...
社区修复:  社区1 → 社区2, 社区3 → 社区2, ...
```

#### 7.3.2 匹配的具体执行

`choose_community_bridges` 调用 `choose_matching_for_size`，其执行过程：

```python
# 构造二分图邻接表
# 左部 = 作为 source 的流索引
# 右部 = 作为 target 的流索引
adjacency = {0: [1, 2], 1: [0, 2], 2: [0, 1]}

# 第 1 步：Hopcroft-Karp 求最大基数
pair_u, _, cardinality = hopcroft_karp(stream_count, adjacency)
# 假设 cardinality = 2（最多能合并 2 对）

# 第 2 步：在基数硬约束下做 min-cost flow
selected_pairs = min_cost_matching_exact_cardinality(
    stream_count, candidates, required_flow=cardinality
)
```

#### 7.3.3 完整示例

3 条开放路径，候选边如下：

| 候选 | source | target | cost 量级 |
|---|---|---|---|
| b₁ | 社区1 | 社区2 | ~0（精确恢复） |
| b₂ | 社区1 | 社区3 | ~10¹⁵（小额误差） |
| b₃ | 社区2 | 社区3 | ~0（精确恢复） |

Hopcroft-Karp 求最大基数：

```text
左部（source）: {0, 1}    （社区1和社区2可以连别人）
右部（target）: {1, 2}    （社区2和社区3可以被连）

最大基数 = 2
方案一：社区1→社区2（精确）, 社区2→社区3（精确）→ 基数=2, 总代价≈0
方案二：社区1→社区3（小额误差）→ 基数=1, 总代价≈10¹⁵

→ 选方案一（基数最大、代价最小）
```

### 7.4 防环删除（_remove_stream_level_cycles）的完整流程

#### 7.4.1 为什么需要防环

社区端点修复基于**金额近似匹配**而非严格相等。如果 A 的尾巴≈B 的头、B 的尾巴≈C 的头、C 的尾巴≈A 的头，三条本来无关的路径可能凭**数值巧合**连成一个虚假的"全账户大闭环"。这在业务上没有意义（用户并不存在这样的交易循环），因此默认不允许。

#### 7.4.2 算法流程

```python
by_source = {b.source_stream_index: b for b in selected}
state = {}  # 0=未访问, 1=正在访问, 2=已回溯

for start in by_source:
    path = []
    position = {}
    current = start
    while current in by_source and state[current] == 0:
        state[current] = 1          # 标记为"正在访问"
        position[current] = len(path)
        path.append(current)
        current = by_source[current].target_stream_index
        
        if current in position:     # 发现环！
            cycle_edges = path[position[current]:] 的修复边
            worst = max(cycle_edges, by cost)
            removed.add(worst)
            break
    
    for node in path:
        state[node] = 2             # 标记为"已回溯"
```

这是一个经典的三色 DFS 找环算法（白色=未访问、灰色=正在访问、黑色=已回溯）。

#### 7.4.3 示例

假设 HK + min-cost flow 选中的 3 条修复边构成了环：

```text
社区1 ──(cost=100)──→ 社区2
  ↑                      │
  │                      │(cost=50)
  │                      ↓
  └────(cost=200)──── 社区3
```

DFS 路径追踪：

```text
从社区1出发: path=[1]          position={1:0}
→ 社区1→社区2: path=[1,2]      position={1:0, 2:1}
→ 社区2→社区3: path=[1,2,3]    position={1:0, 2:1, 3:2}
→ 社区3→社区1: current=1 已在 position 中！
   cycle_sources = [1, 2, 3]
   cycle_edges 的 cost: [100, 50, 200]
   worst = max(100, 50, 200) = 社区3→社区1（cost=200）
```

删除社区3→社区1（最不可信的边），保留精确的社区1→社区2（cost=50）：

```text
应用：社区1→社区2（保留）, 社区2→社区3（保留）
社区3→社区1（已删除）
→ 仍为开放路径，没有形成环
```

### 7.5 合并应用（apply_community_bridges）的完整流程

经过选择的修复边列表（已去除环边）被应用到 successor 图：

```python
# 为快速检查构建 predecessor 字典
predecessor = {target: source for source, target in successor.items()}

for bridge in selected:
    # 防御校验
    assert bridge.source_node not in successor  # 末节点不能已有后继
    assert bridge.target_node not in predecessor  # 首节点不能已有前驱
    
    # 追加修复边
    successor[bridge.source_node] = bridge.target_node
    predecessor[bridge.target_node] = bridge.source_node
    
    # 修正目标节点的余额影响额
    inferred_effects[bridge.target_node] = bridge.corrected_effect_minor
```

#### 7.5.1 合并前后的 successor 变化

合并前：

```text
successor = {
    A的索引: B的索引,        # 社区1内部
    B的索引: C的索引,
    D的索引: E的索引,        # 社区2内部
    F的索引: G的索引,        # 社区3内部
    G的索引: H的索引,
}
```

选中的修复边：社区1→社区2（`bridge.source_node=C的索引, bridge.target_node=D的索引`）

合并后：

```text
successor = {
    A的索引: B的索引,        # 社区1内部
    B的索引: C的索引,
    C的索引: D的索引,        # ← 追加的修复边！C→D
    D的索引: E的索引,        # 社区2内部
    F的索引: G的索引,        # 社区3内部（未受影响的独立社区）
    G的索引: H的索引,
}
```

#### 7.5.2 重新分解后的结果

再次调用 `decompose_streams(successor)`：

```text
构建 predecessor:
  B←A, C←B, D←C, E←D, G←F, H←G

找到无前驱的起点: A(无前驱), F(无前驱)

从 A 出发: A → B → C → D → E  (一条路径，合并了社区1和社区2)
从 F 出发: F → G → H           (另一条路径，社区3保持独立)

最终: 2 条路径（原来 3 条），完成了合并
```

### 7.6 余额影响额的修正

#### 7.6.1 为什么需要修正

合并前的严格匹配基于 `after[i] == before[j]`。合并后，社区1的末节点 C（after=100）需要连接到社区2的首节点 D（before=D.after - E_t）。

合并前 D 的 `inferred_effects[D] = 50`，所以 `before[D] = 150 - 50 = 100`。
而 after[C] = 100，所以 `after[C] == before[D]` 成立。

但如果 D 的原始余额影响 E_t 是 -30（方向写反），那么 `before[D] = 150 - (-30) = 180`，而 `after[C] = 100`，不相等。

#### 7.6.2 修正的计算

```python
desired_effect = B_t - B_s = 150 - 100 = 50
inferred_effects[D] = 50  # 从 -30 修正为 50
```

修正后：
- `before[D] = 150 - 50 = 100`
- `after[C] = 100`
- **余额连续性成立**

#### 7.6.3 修正的物理含义

| 概念 | 值 | 含义 |
|---|---|---|
| 原始金额 `amount_minor` | 30 | CSV 中写的金额（不修改） |
| 原始方向 `direction` | "出" | CSV 中写的方向（不修改） |
| 原始影响 `effective_minor` | -30 = -1×30 | 按原始方向计算的余额影响 |
| 修正影响 `inferred_effects` | 50 | 实际上要让路径连续所需的影响 |
| SVF 显示的修正金额 | 50÷1×(-1) = -50 | 从修正影响反推回金额，橙色/紫色标记 |

注意：原始 CSV **不被覆盖**。所有修正只体现在：
- `inferred_effects` 数组（运行时内存数据）
- 输出 CSV 的 `__corrected_signed_amount`、`__inferred_effect_amount` 列
- SVG 图中的橙色（金额修复）或紫色（符号修复）标记

### 7.8 小结

社区端点修复算法的核心设计思想：

| 设计点 | 目的 | 实现方式 |
|---|---|---|
| **搜索优化** | 避免 O(n²) 暴力枚举 | 余额索引 + 二分查找 |
| **双层去重** | 避免冗余候选 | per_target 去重 → 全局去重 |
| **三层代价编码** | 确保优先级正确 | 类别×10¹⁵ + 误差×10⁹ + 时间×10⁸ |
| **匹配框架** | 全局最优而非局部最优 | HK + min-cost flow |
| **防环机制** | 防止数值巧合形成虚假闭环 | 三色 DFS 删最弱边 |
| **修正不覆盖原值** | 数据可追溯 | 只改 inferred_effects，不改 Transaction 原始字段 |

## 8 环断开与重连（break_cycles_into_paths）

### 8.1 动机

两阶段匹配（HK + min-cost flow）以最大化匹配基数优先，在相同基数下最小化总代价。对于 test4.csv 这类数据，匹配结果的代价最优解会产生一个环（3→4→5→6→3）加一条路径（1→2），而不是单一哈密尔顿路径（1→3→4→5→6→2）。这是因为：

| 边 | 时间差 | 行号差 | 代价 | 比较 |
|---|---|---|---|---|
| 1→2 | 98 秒 | 1 | ≈2 | ★ 很低 |
| 1→3 | 94306 秒 | 2 | ≈741 | 较高 |
| 5→2 | 负时间差 | -4 | ≈1214787 | ★ 很高 |
| 5→3 | 负时间差 | -3 | ≈1214777 | 略低 |

匹配选择了 1→2（代价极低）+ 闭合环 3→4→5→6→3（5→3 比 5→2 略低），总代价小于 1→3 + 5→2。环断开与重连作为**后处理步骤**，在不改变匹配框架的前提下，将环断开并交叉重连为单一路径。

### 8.2 算法全流程

```text
输入: successor（有向图），transactions，edges（候选边代价）
输出: 断开的环数

────────────────────────────────────────────────────
第一阶段：找环并断开
────────────────────────────────────────────────────
  对每个未访问的节点 start：
    沿 successor 指针追踪路径
    若发现当前节点已在路径中 → 找到环！
      找出环中所有边
      找到代价最高的边（最不可信）
      从 successor 中删除该边（断开环）
      环变为开放路径
    标记路径上所有节点为已访问

  若没有断开任何环 → 直接返回 0

────────────────────────────────────────────────────
第二阶段：交叉重连（最多 3 轮）
────────────────────────────────────────────────────
  每轮迭代：
    重建前驱表 predecessor
    找出所有路径起点（无前驱的节点）
    对每条路径，追踪到终点 → 得到 paths = {(s1→e1), (s2→e2), ...}

    对每对路径 (s1→e1) 和 (s2→e2)：
      模式A（标准合并）：
        if after[e1] == before[s2]:
          断开 e1 原有后继（若有）
          断开 s2 原有前驱（若有）
          连接 e1 → s2
          → 两条路径合并为一条

      模式B（交叉合并）：
        if after[e2] == before[e1] AND after[s1] == before[s2]:
          断开 e2 原有后继
          断开 e1 原有前驱
          断开 s1 原有后继
          断开 s2 原有前驱
          连接 e2 → e1
          连接 s1 → s2
          → 两条路径交叉合并为一条

    若本轮没有成功连接 → 结束迭代
```

### 8.3 第一阶段：三色 DFS 找环

使用经典的三色 DFS（深度优先搜索）在 `successor` 构成的有向图中检测环：

| 颜色 | 值 | 含义 |
|---|---|---|
| 白色（0） | 未访问 | 节点尚未被遍历 |
| 灰色（1） | 正在访问 | 节点在当前遍历路径中 |
| 黑色（2） | 已回溯 | 节点的所有后继已遍历完毕 |

```python
visited = [0] * n
for start in range(n):
    if visited[start] != 0:
        continue
    cur = start
    while cur != -1 and visited[cur] == 0:
        visited[cur] = 1           # 标记为灰色
        pos[cur] = len(path)        # 记录在路径中的位置
        path.append(cur)
        cur = successor.get(cur, -1)
        if cur in pos:              # 发现环！
            cycle_nodes = path[pos[cur]:]  # 提取环中的节点
```

**举例**：test4.csv 匹配后的 successor = `{0→1, 2→3, 3→4, 4→5, 5→2}`

```text
遍历顺序：
  start=0: 0→1(-1), 结束。无环。标记 0,1 为黑色
  start=1: 已访问，跳过
  start=2: 2→3→4→5→2，发现 2 已在路径中！
    环中节点: [2,3,4,5]
    环中边:   2→3(cost=738), 3→4(cost=1), 4→5(cost=1), 5→2(cost=1214787)
    最贵边:  5→2(cost=1214787) → 断开！
    successor 变为: {0→1, 2→3, 3→4, 4→5}
```

### 8.4 最贵边选择

对环中的每条边 `(src→tgt)`，从候选边列表中查找其代价：

```python
c = edge_cost_map.get((src, tgt), 10**9)  # 默认极高代价
if worst_edge is None or c > worst_edge[2]:
    worst_edge = (src, tgt, c)
```

选择代价最高的边断开，基于以下考虑：
- 代价高的边最不可信（时间差大、行号差远、方向冲突）
- 断开它最不可能破坏真实的业务关系

### 8.5 第二阶段：两种重连模式

断开环后，`successor` 中只包含路径。第二轮的任务是将多条路径合并为更少的路径。

#### 模式A：标准端→首合并

```text
路径1: s1 → ... → e1
路径2: s2 → ... → e2

条件: after[e1] == before[s2]
操作: e1 → s2（同时断开 e1 原有后继、s2 原有前驱）

结果: s1 → ... → e1 → s2 → ... → e2  （一条路径）
```

#### 模式B：交叉合并

当路径的端点与另一条路径的端点无法直接端→首连接时，尝试交叉连接：

```text
路径1: s1 → ... → e1
路径2: s2 → ... → e2

条件: after[e2] == before[e1] 且 after[s1] == before[s2]
操作: e2 → e1，s1 → s2（同时断开所有冲突边）

结果: s1 → s2 → ... → e2 → e1  （一条路径，路径2被嵌入路径1中间）
```

### 8.6 test4.csv 的完整推演

**初始状态**（匹配后）：

```text
successor = {0→1, 2→3, 3→4, 4→5, 5→2}
路径1: 0→1  (start=0, end=1, after=95)
环:    2→3→4→5→2
```

**第 1 步：找环并断开**

```text
发现环 [2,3,4,5]，最贵边 5→2(cost=1214787)
断开 5→2
successor = {0→1, 2→3, 3→4, 4→5}
路径1: 0→1  (end=1, after=95)
路径2: 2→3→4→5  (start=2, end=5, after=100)
```

**第 2 步：检查模式A**

```text
检查 after[e1=1]=95 == before[s2=2]=100? → 95≠100，不匹配
检查 after[e2=5]=100 == before[s1=0]=0? → 100≠0，不匹配
→ 模式A无法合并
```

**第 3 步：检查模式B**

```text
检查 after[e2=5]=100 == before[e1=1]=100? → 100==100 ✓
检查 after[s1=0]=100 == before[s2=2]=100? → 100==100 ✓
→ 模式B匹配！

操作:
  删除 e2=5 原有后继（无）
  删除 e1=1 原有前驱（0→1）
  删除 s1=0 原有后继（0→1）
  删除 s2=2 原有前驱（无）
  添加 e2→e1: successor[5] = 1
  添加 s1→s2: successor[0] = 2

结果 successor = {2→3, 3→4, 4→5, 5→1, 0→2}
路径: 0→2→3→4→5→1  （一条哈密尔顿路径！）
```

### 8.7 关键设计要点

| 要点 | 说明 |
|---|---|
| **后处理** | 不改变原有匹配框架，在匹配完成后独立执行 |
| **三色 DFS** | O(n) 时间检测所有有向环 |
| **最贵边优先** | 断开最不可信的边，保留最可信的连接 |
| **模式A** | 标准端→首合并，一次连接合并两条路径 |
| **模式B** | 交叉合并，处理端→首无法匹配但交叉匹配成立的情况 |
| **多轮迭代** | 最多 3 轮，每轮合并一对路径 |

### 8.8 与链端点容错合并的区别

| 特性 | 链端点容错合并 (S5) | 环断开重连 (break_cycles) |
|---|---|---|
| **触发条件** | 金额误差 < 阈值（默认 ¥1.00） | 存在有向环 |
| **匹配对象** | 开放路径的首尾节点 | 图中的所有路径端点 |
| **允许误差** | 是（小额误差/符号修复） | 否（严格相等） |
| **重连方式** | 在 HK+flow 框架中做最大基数匹配 | 直接检查 after==before |
| **防环** | 三色 DFS 删除环中最贵边 | 本身就是在断环 |
| **代价计算** | `class_rank×10¹⁵+误差×10⁹+时间×10⁸` | 直接使用候选边代价 |
